## ES 在整个项目中的定位

```
用户搜索 → 网关 → 搜索服务 → 优先查 ES（毫秒级）
                          → ES 挂了自动查 MySQL（秒级，兜底）
```

灵感数据的主存储是 **MySQL**，ES 只是一个**搜索加速层**。灵感是通过 `inspire-core` 服务写入 MySQL 的，ES 里的数据是 MySQL 的副本。

## 数据怎么进 ES

有两种方式，目前都还没做，但在 PRD 里有规划：

**方式一：Canal 实时同步（PRD 方案）**
```
MySQL binlog → Canal → ES
```
灵感发布到 MySQL 后，Canal 监听到 binlog 变更，自动同步到 ES 的 `inspire_index`。
这是生产环境的标准方案，延迟在秒级。`middlewareDoc.md` 里已经把 Canal 的配置写好了。

**方式二：手动同步（当前开发阶段）**
由于你本地不一定有 Canal，可以先手动把数据灌进 ES：

```bash
# 从 MySQL 查一条灵感，写入 ES
curl -X PUT "http://localhost:9200/inspire_index/_doc/{id}" \
  -H "Content-Type: application/json" \
  -d '{...灵感数据JSON...}'
```

**方式三：代码同步（后续可加）**
在 `inspire-core` 的发布接口里，发布成功后同步调 ES 的 API 写入。简单直接，但耦合了。

## ES 搜索的流程

当用户搜索 `"鸡腿"`：

```
1. 搜索服务收到请求，mode=auto
2. 调 ES：GET /inspire_index/_search
   {"query":{"multi_match":{"query":"鸡腿","fields":["title^10","tag^5"]}}}
   → title 匹配权重 10，tag 匹配权重 5
3. ES 返回命中文档（毫秒级）
4. 如果 ES 超时/连不上 → 自动降级 MySQL LIKE
   SELECT ... WHERE title LIKE '%鸡腿%' OR tag LIKE '%鸡腿%'
```

## 为什么 ES 比 MySQL LIKE 快

| 对比 | MySQL LIKE | ES |
|------|-----------|----|
| 全表扫描 | ✅ 必须扫全表 | ❌ 倒排索引直接定位 |
| 中文分词 | ❌ 只能 %关键词% 模糊 | ✅ 支持 ik 分词器 |
| 相关度排序 | ❌ 只能按固定字段 | ✅ TF-IDF/BM25 算法 |
| 百万数据 | 秒级 | 毫秒级 |
| 高亮 | ❌ 需自己拼 | ✅ 内置 highlight |

## 目前搜索服务的架构

```
SearchController
  ↓
SearchServiceManager  ← 配置 inspire.search.mode
  ├── mode=auto → 先调 EsSearchService，异常时降级 MysqlSearchService
  ├── mode=es   → 只调 EsSearchService，ES 挂了就报错
  └── mode=mysql → 只调 MysqlSearchService，不需要 ES
```

**EsSearchService**：
- `RestClient` 懒加载（启动时不连 ES，第一次搜索才连）
- 从 `inspire.es.hosts` 配置读 ES 地址（默认 `localhost:9200`）
- 用低级 REST API 拼接查询，不依赖 Spring Data

**MysqlSearchService**：
- 直接用 `JdbcTemplate` 执行 `LIKE` 查询
- 不依赖 MyBatis-Plus，轻量兜底
